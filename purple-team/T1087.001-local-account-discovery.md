## What Is T1087.001?

After gaining initial access to a system, one of the first things an attacker does is enumerate who is on the machine and what privileges they have. T1087.001 covers Local Account Discovery — the techniques used to identify user accounts on a compromised Linux or Windows host.

On Linux, this typically involves reading `/etc/passwd`, running `id`, `groups`, `whoami`, and `lastlog` to build a picture of the user's identity, group memberships, and privilege level. This information directly informs the attacker's next move — specifically, whether privilege escalation is possible or necessary.

## Attack Output Analysis

### Test 5 — Remote Login History
```
sh: 1: lastlog: not found
Exit code: 0
```
**Attacker interpretation:** `lastlog` is not installed on this system. This tells the attacker the machine may be a minimal or purpose-built install. No remote login history is available — a minor intelligence gap, but the exit code 0 means execution was not blocked.

### Test 6 — Enumerate Users and Groups (Most Critical)

**Raw output:**
```
kali adm dialout cdrom floppy sudo audio dip video plugdev users netdev scanner bluetooth lpadmin wireshark kaboxer
uid=1000(kali) gid=1000(kali) groups=1000(kali),4(adm),20(dialout),24(cdrom),25(floppy),27(sudo),29(audio),30(dip),44(video),46(plugdev),100(users),101(netdev),102(scanner),104(bluetooth),113(lpadmin),122(wireshark),123(kaboxer)
```

**Attacker interpretation, line by line:**

| Finding | Value | Attacker Significance |
|---|---|---|
| Username | `kali` | Default Kali install user confirmed |
| UID | `1000` | Standard user, not root |
| GID | `1000` | Primary group is kali |
| `sudo` group | `27(sudo)` | **Critical** — user can escalate to root |
| `adm` group | `4(adm)` | Can read system logs in `/var/log` |
| `wireshark` group | `122(wireshark)` | Can capture network traffic without root |
| OS fingerprint | Default Kali groups present | Confirms OS and default configuration |

**Attacker next steps based on this output:**
1. Run `sudo -l` to enumerate available sudo commands
2. Attempt privilege escalation to root via sudo
3. Read `/var/log/auth.log` using adm group access to review authentication history
4. Use wireshark group access to capture network traffic for credential sniffing

Two commands. Five seconds. Full privilege roadmap identified.

## Detection Opportunity (Future Wazuh Rule)

When Seraph comes online with Wazuh, the following behavioral pattern should trigger an alert:

**Detection logic:**
- `id`, `groups`, `whoami`, `cat /etc/passwd`, or `lastlog` executed in rapid succession
- Parent process is a shell (`bash`, `sh`, `pwsh`) rather than a legitimate application
- Execution occurs outside of normal working hours or from an unusual user session

**Sigma rule concept:**
```yaml
title: Local Account Enumeration - Linux
status: experimental
description: Detects rapid sequential execution of account enumeration commands
logsource:
  category: process_creation
  product: linux
detection:
  selection:
    CommandLine|contains:
      - 'cat /etc/passwd'
      - 'cat /etc/sudoers'
      - '/bin/id'
      - 'groups'
      - 'lastlog'
  condition: selection
falsepositives:
  - System administrators performing routine checks
level: medium
tags:
  - attack.discovery
  - attack.t1087.001
```

This rule will be implemented on Seraph post-build. T1087.001 will be re-executed via CALDERA agent on Sentinel (the dedicated target machine), with Wazuh watching in real time. VECTR outcome will be updated from **None** to reflect actual detection capability.

## Remediation Notes

For a production environment, the following hardening steps would reduce the impact of this technique:

- Implement auditd rules to log execution of enumeration commands
- Alert on `sudo` group membership queries from unexpected processes
- Restrict `/etc/passwd` and `/etc/sudoers` read access where possible
- Deploy a SIEM (Wazuh/Splunk) with process execution logging enabled
